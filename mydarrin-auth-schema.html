Schema Conturi & Autentificare în 2 Pași

Acest document definește structura logică (scheme JSON) pentru toate tipurile de conturi de pe platforma My Darrin, pregătite pentru implementarea unui backend real. Important: acest fișier descrie arhitectura țintă — implementarea curentă a platformei (HTML static) simulează vizual fluxul, fără persistență reală sau verificare OTP funcțională.

🔐 Flux unic de înregistrare — 2 pași, pentru toate rolurile

1
Email/Telefon + Parolă
Utilizatorul introduce identificator (email sau telefon) + parolă. Backend creează contul cu status: "pending_otp" — fără acces la platformă încă.
2
Cod OTP (SMS sau Email)
Cod 6 cifre, valabil 10 minute, trimis către telefonul/emailul declarat. La validare corectă, status: "active" — abia acum contul e funcțional.
// Obiect comun tuturor rolurilor — câmpurile de bază ale autentificării { "account_id": "uuid-v4", "role": "client" | "partner" | "investor", "auth": { "identifier": "email@exemplu.ro" // email SAU telefon, "identifier_type": "email" | "phone", "password_hash": "bcrypt(...)", "status": "pending_otp" | "active" | "suspended", "otp": { "code_hash": "sha256(...)", "expires_at": "ISO-8601", "attempts": 0, "channel": "sms" | "email" }, "created_at": "ISO-8601", "last_login_at": "ISO-8601 | null" } }

👤 Rol: Client

Un singur tip de cont, diferențiat prin client_type — nu necesită conturi separate pentru B2C/B2B/B2G.

B2C — Persoană fizică B2B — Companie B2G — Instituție publică
{ "role": "client", "client_type": "B2C" | "B2B" | "B2G", "profile": { "nume": "...", "prenume": "...", "companie": "... | null" // obligatoriu pentru B2B/B2G, "cui_cif": "... | null", "cod_entitate_publica": "... | null" // specific B2G, "adresa_facturare": { "judet": "...", "oras": "...", "strada": "..." } }, "comenzi_active": ["... referințe order_id"] }

🤝 Rol: Partener — 5 sub-tipuri

Toate cele 5 roluri folosesc același obiect de bază (role: "partner"), diferențiate prin partner_type — consistent cu PARTNER_META deja definit în mydarrin-devino-partener.html.

Schemele de mai sus definesc arhitectura țintă. Platforma curentă (fișiere HTML statice) simulează vizual fluxurile de cont (sessionStorage, fără backend), motiv pentru care nu există încă: verificare OTP funcțională, persistență de cont între sesiuni de browser, sau cale de "logare" pentru un utilizator existent. Implementarea reală a acestei scheme necesită un backend (similar sistemului Vercel KV deja construit pentru branding) conectat la endpoint-uri /api/auth/register și /api/auth/verify-otp.
